Foundation

01Why Docker Networking Exists

Containers are isolated processes. By default, they have no way to talk to the outside world or to each other. Docker Networking is the subsystem that gives containers internal and external connectivity in a controlled, configurable way.

Think of it exactly like a VPC on AWS — just as EC2 instances inside a VPC communicate over private IPs, containers inside a Docker network communicate over Docker-assigned private IPs.

💡 Role Clarity in DevOps

Your job as a DevOps engineer is to write Dockerfiles and build images. Running and orchestrating containers at scale is Kubernetes' job. Docker networking concepts become the foundation for understanding Kubernetes pod networking, CNI plugins, and services — so learning this well pays off later.

02The Three Network Types

Docker ships with three built-in network drivers. Every container must be on exactly one of these (or a custom variant of bridge):

🌉

BRIDGE

The default. Creates a private internal network on the host. Containers get their own IPs and talk through a gateway. Port mapping required for external access.

DEFAULT
🖥

HOST

Container shares the EC2's network stack directly. No isolation, no port mapping, no overhead. Container uses host ports directly. Risk: port conflicts.

HIGH PERFORMANCE
🔒

NONE

Container has NO network interface. Completely isolated — cannot reach the internet, other containers, or the host. Maximum isolation.

FULLY ISOLATED
📌 Quick Check — from docker network ls

Run docker network ls and you will always see these three pre-created: bridge (driver: bridge), host (driver: host), none (driver: null). Any networks you create yourself will also appear here.

$ docker network ls

Network Type 1

03Bridge Network — The Default

When you run docker run nginx without specifying a network, Docker attaches the container to the default bridge network. This creates a virtual switch inside your EC2 host.

How Bridge Networking Works Internally

Docker creates a virtual network interface called docker0 on the host. This acts as the gateway (172.17.0.1). Every container that joins the bridge gets an IP from the 172.17.0.0/16 subnet — each container gets a unique IP in this range.

🌉 Bridge Network Architecture — Your Actual Lab

Key Properties of Bridge Network

From your lab, the docker inspect output confirmed exactly this architecture:

  • Gateway172.17.0.1 — this is the docker0 virtual interface on the EC2 host
  • nginx #1172.17.0.2 (no port mapping — internal only)
  • nginx #2172.17.0.3 (mapped EC2:81 → container:80)
  • httpd172.17.0.4 (mapped EC2:80 → container:80)
  • Subnet172.17.0.0/16 — can hold up to 65,534 containers

Container-to-Container Communication (Ping Test)

Your lab proved that all containers on the same bridge network can ping each other using their private IPs — without any port mapping. The Ubuntu container (172.17.0.5) successfully pinged the httpd container (172.17.0.4) with zero packet loss.

Inside Ubuntu container on bridge network:

⚠️ Default Bridge Limitation

The default bridge network does NOT support DNS-based container discovery. You can only ping by IP. To ping by container name (e.g., ping apache-server), you need a custom bridge network — which Docker creates when you run docker network create. Custom bridges have built-in DNS resolution.

Port Mapping — -p vs -P

Lowercase -p: you choose the port mapping 
docker run -dt -p 81:80 nginx
# EC2 port 81 → container port 80 (you control which EC2 port)
# Access via: http://<ec2-ip>:81
Uppercase -P: Docker chooses a random high port 
docker run -dt -P nginx
PORTS: 0.0.0.0:32768->80/tcp
# Docker randomly assigned port 32768 on EC2 → container port 80
# Useful in dev/testing when you don't care which port is used
Network Type 2

04Host Network — Sharing EC2's Network Stack

With --network host, the container does NOT get its own network namespace. It directly uses the EC2 instance's network interfaces, IPs, and ports. There is no docker0 gateway, no NAT, no port mapping.

🖥 Host Network — Container Shares EC2 Network Stack
Your exact lab output proving the conflict: 
# yum-installed nginx already holding port 80
docker run -dt --network host --name rakesh nginx
docker logs 3f21e55c3ab8

[emerg] 1#1: bind() to 0.0.0.0:80 failed (98: Address already in use)
nginx: [emerg] bind() to [::]:80 failed (98: Address already in use)
nginx: [emerg] still could not bind()

# After yum remove nginx → no more conflict → works!
docker run -dt --network host nginx
[notice] 1#1: nginx/1.29.8
[notice] 1#1: start worker processes  ← SUCCESS

# -p flag is silently IGNORED with host network:
docker run -dt -p 81:80 --network host nginx
WARNING: Published ports are discarded when using host network mode
🚨 Host Network Pitfalls

Port conflicts — if EC2 runs any service on the same port, the container will fail to start. Always check with ss -tuln before running host-network containers. No isolation — any port the container opens is immediately exposed on the EC2 IP. And no multiple replicas — you cannot run two host-network containers that both want port 80.

✅ When to Use Host Network

Host networking gives maximum throughput with zero NAT overhead. Use it for high-performance scenarios like network monitoring tools, packet sniffers, or when the container needs to listen on many dynamic ports. Best practice: dedicate the EC2 to Docker containers only — don't install application services (nginx, mysql) directly on the EC2 OS.

Network Type 3

05None Network — Complete Isolation

With --network none, the container gets its own isolated network namespace but no network interfaces are created inside it except the loopback (lo). No gateway. No IP. Cannot reach anything.

🔒 None Network — Fully Isolated Container
docker inspect proves no IP, no gateway: 
docker run -dt -p 81:80 --network none nginx
# Inspect NetworkSettings:
"Networks": {
    "none": {
        "Gateway": "",        ← no gateway
        "IPAddress": "",     ← no IP assigned
        "MacAddress": "",    ← no MAC address
        "NetworkID": "cc7ed4a97227..."
    }
}
💡 When is "none" useful?

Security-sensitive processing jobs — for example, a container that processes sensitive documents (PII data, cryptographic operations) where you want guaranteed network isolation. Also useful for batch jobs that read from mounted volumes and write output locally with no need for any network.

Network Type 4

06Custom Bridge Networks

You can create your own bridge networks with docker network create <name>. The benefit over the default bridge: automatic DNS resolution by container name and network isolation between groups of containers.

🔀 Custom Networks — Isolation Between Teams / Environments
Creating and using a custom network: 
# Create custom network named "dev"
docker network create dev
f3631353aeff2f77a0c6fc099567ab2ba9922a7e792ceafbb88a45381fd656bd

# Run a container in the custom network
docker run -it --network dev ubuntu

# This container gets 172.18.0.2 — a DIFFERENT subnet from bridge (172.17.x.x)
# Ping from dev network to bridge network FAILS — correct! Networks are isolated.
root@2156cea3c683:/# ping 172.17.0.4
# No response → cross-network isolation working as designed

Here's the exact path a request takes to reach your containerized application in a bridge network setup:

Industry Practice

09Real-World Scenarios

1

Microservices App — Frontend + Backend + DB

A typical 3-tier app has a React frontend, Node.js API, and MongoDB. All three run as containers. You create one custom bridge network ("app-net"). All containers join it. The API talks to MongoDB by name (mongodb:27017) using Docker DNS. Only the frontend is port-mapped to EC2 for external access.

docker network create app-net
docker run -d --network app-net --name mongodb mongo
docker run -d --network app-net --name api -e MONGO_URL=mongodb://mongodb:27017/db my-api
docker run -d --network app-net -p 80:3000 my-frontend
# API resolves "mongodb" by name → Docker DNS → 172.18.0.2
2

CI/CD Pipeline — Isolated Build Environments

In a Jenkins pipeline, each build job runs in a none or custom bridge network container to prevent builds from interfering with each other. Security-critical builds (processing secrets) use --network none to ensure no exfiltration is possible.

# Secure build: no network access — reads from volume, writes to volume
docker run --network none -v /build:/workspace my-build-image
# Test environment: isolated bridge so tests don't hit prod
docker run --network test-net my-test-runner
3

High-Performance Proxy/Load Balancer — Host Network

When running nginx as a reverse proxy/load balancer in front of many services, host networking eliminates NAT latency. The EC2 must be dedicated to this proxy container only. No other services should run on port 80/443.

# EC2 is dedicated — no system nginx, no other apps on 80/443
docker run -d --network host --restart always nginx-proxy
# Result: same throughput as bare-metal nginx, zero NAT overhead
4

Dev/Staging/Prod on One EC2 (Cost Saving)

Three separate custom networks isolate environments. Containers in "dev-net" cannot accidentally reach containers in "prod-net". Each network gets its own IP range via Docker's IPAM.

docker network create dev-net
docker network create staging-net
docker network create prod-net
docker run --network dev-net my-app:dev     # 172.18.x.x
docker run --network staging-net my-app:staging# 172.19.x.x
docker run --network prod-net my-app:prod    # 172.20.x.x
🎯 Real-World Truth from Your Trainer

In real production, you rarely manage Docker networks and volumes manually — Kubernetes handles all of this. Docker's role ends at building the image and pushing it to a registry. K8s takes the image and handles networking, scaling, and storage. However, understanding Docker networking deeply is essential because Kubernetes networking (pods, services, CNI) follows the same fundamental concepts.

10Command Cheat Sheet

docker network ls List all networks (bridge, host, none, custom)
docker network create <name> Create a custom bridge network
docker run --network bridge nginx Explicit bridge (same as default)
docker run --network host nginx Use EC2's network stack directly
docker run --network none nginx Fully isolated — no network access
docker run --network dev nginx Join a custom user-created network
docker run -p 81:80 nginx Map EC2:81 → container:80 (your choice)
docker run -P nginx Map random high port → container exposed ports
docker inspect <id> Show IP, Gateway, NetworkSettings of container
docker network inspect <name> Show all containers on a network with IPs
docker network connect <net> <ctr> Add running container to an additional network
docker network rm <name> Delete a custom network (containers must be removed)
docker system prune Remove stopped containers, unused networks, dangling images
ss -tuln Check which ports are in use on EC2 (avoid conflicts)
🧠 Mental Model to Remember

Think of Docker networking like a building with floors: Bridge = separate apartments with a shared lobby (gateway) — you need a room number (port mapping) to be reachable from outside. Host = open office with no walls — everything is visible to everyone, but you can't have two people on the same phone extension. None = soundproof vault — completely isolated, nothing goes in or out. Custom bridge = a private floor only your team has access to, with intercom (DNS) by name.
